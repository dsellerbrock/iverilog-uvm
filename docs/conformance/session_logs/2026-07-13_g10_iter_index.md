# Session log — 2026-07-13 (fourth target): 7.12.4 iterator index querying

Engineering target: IEEE 1800-2017 **7.12.4 Iterator index querying**
— `function int_or_index_type index (int dimension = 1)` on array
method iterators.  The last gap-audit-cited 7.12 breakage
(`Variable item does not have a field named: index.`).

## LRM grounding (clause text previously extracted from the 1800-2017 PDF)

"The index method of an iterator returns the index value of the
specified dimension... If the dimension is not specified, the first
dimension is used by default.  The return type of the index method is
an int for all array iterator items except associative arrays."
Literal example: `q = arr.find with ( item == item.index );`

## Implemented

- **Iterator context stack** (elab_expr.cc; API in netmisc.h:
  `push_array_method_iter_ctx` / `pop_array_method_iter_ctx` /
  `find_array_method_iter_index`): every loop builder already
  allocates a loop-counter net; while a with expression elaborates,
  the (iterator net → idx net) pair is pushed.  A stack because with
  expressions nest — lookups match by iterator net pointer, so an
  outer iterator's index remains reachable from inside an inner with
  expression, and inner shadows outer for the default `item` name.
  Wired in all four builders: `elab_array_method_with_expr_`
  (reductions, min/max), `make_queue_locator_with_expr_` (find*),
  and the sort_with elaboration in elaborate.cc.
- **Interceptions**: PEIdent (paren-less `item.index`, just before
  the "does not have a field named" error) and
  `PECallFunction::elaborate_expr_method_` (`item.index()` /
  `item.index(1)`; any other dimension is an explicit sorry citing
  7.12.4).  Both resolve to a NetESignal of the s32 idx net.  Outside
  a with expression `item.index` remains an error (no leakage).
- **test_width repair (broader than the target)**: the nested probe
  `q.sum(o) with (zq.sum(x) with (x.index) + o.index)` returned 0 —
  the emitted code padded both operands with `%pad/s 0`.  Root cause:
  PECallFunction::test_width reported width 0 for array reduction
  methods and PEIdent::test_width width 0 for `item.index`, so the
  containing binary node computed context width 0 and elaboration
  padded the operands to ZERO BITS.  This affected reductions in ANY
  operand position — `r = q.sum() + 1` evaluated to 0 before this
  fix.  test_width_method_ now reports the element type's
  width/signedness for with-less reductions (verified by the signed
  byte case: `b.sum() + 0` gives -127) and a 32-bit signed
  approximation when a with clause supplies the values (only ever
  widens — the exact width is applied at elaboration); both
  interception points report int for the iterator index.

## Verified (permanent test `tests/g10_iter_index_test.sv`, 19 checks)

The 7.12.4 LRM literal example; find_index/sum/max/sort with index
predicates and keys; index()/index(1) call forms; custom iterator
names; class-property receivers; nested with expressions (inner
index, and outer index from inside the inner with); reductions in
operand contexts (plain, with-clause, mixed, signed element width).

## Remaining tail (gap audit updated)

`item.index` on fixed arrays with non-zero-based declared ranges (the
loop counter is the canonical 0-based address) and dimension > 1 —
both explicit sorry/diagnostic paths, no silent misbehavior.
